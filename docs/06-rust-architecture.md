# Rust Architecture

## Dependency rule

The combat crate must not depend on Bevy or another engine.

```text
Excel .xlsx --Sora--> generated Rust + config.sora
                                      |
                                 combat-data
                                      |
                           +----------+----------+
                           v                     v
                      combat-core <-------- activity-core
                           ^                     ^
                           |                     |
                    content-rules      mode-standard / mode-challenge /
                           |           mode-universe / mode-event
                           +----------+----------+
                                      |
                  combat-ai / combat-replay / sim-cli / engine adapters
```

`combat-core` owns exactly one battle: units, formulas, timeline, actions, effects, enemies, waves, events, and battle RNG. `activity-core` owns every cross-battle concern: flow graph, scoped variables, participants/loadouts, decisions, clocks, scores, objectives, persistence, BattleSpec/Result handoff, and activity hashing. Standard battles, rotating challenges, Simulated Universe families, and future events are mode profiles over `activity-core`; they do not introduce new generic state machines. `content-rules` is the static registry for exceptional battle/activity handlers and cannot bypass normal operations or events. `combat-data` converts Sora-generated records into validated immutable combat and activity catalogs.

Production entry points construct even an ordinary encounter as a one-Battle-node activity. Formula tests and low-level tools may construct `Battle` directly. Excel is an authoring input only; core crates and engine adapters never read workbooks.

The normative configuration workflow is defined in [Excel and Sora configuration pipeline](07-configuration-pipeline.md).

The current repository is in a documentation-only milestone. When executable work begins, create these crate boundaries before content import so generated data, battle rules, activity orchestration, mode profiles, exceptional handlers, and adapters cannot collapse into one package.

All handwritten Rust follows [Rust engineering standards](08-engineering-standards.md). In particular, files are split by responsibility before they exceed 1,200 physical lines, visibility defaults to private, and `pub use` is reserved for a small intentional facade or a documented technical requirement.

## Public core API

Keep mutation behind commands:

```rust
pub struct Battle {
    state: BattleState,
    rng: BattleRng,
}

pub enum Command {
    StartBattle,
    UseAbility {
        actor: UnitId,
        ability: AbilityId,
        primary_target: Option<UnitId>,
    },
    UseInterrupt {
        actor: UnitId,
        ability: AbilityId,
        primary_target: Option<UnitId>,
    },
    PassInterruptWindow,
}

pub struct Resolution {
    pub events: Vec<BattleEvent>,
    pub next_decision: DecisionPoint,
    pub state_hash: [u8; 32],
}

impl Battle {
    pub fn apply(&mut self, command: Command) -> Result<Resolution, CommandError>;
    pub fn view(&self) -> BattleView<'_>;
}
```

`CommandError` is reserved for a rejected command and guarantees no mutation. Accepted commands return a `Resolution`, including a committed `Faulted` transition if an internal deterministic fault occurred. Do not expose mutable unit collections. A controller receives a read-only view plus legal commands for the current decision point.

The activity API mirrors this boundary: `Activity::apply(ActivityCommand) -> Result<ActivityResolution, ActivityCommandError>`. A resolution carries ordered activity events, the next decision, an optional immutable `BattleSpec`, and a canonical activity hash. Submitted `BattleResult` values must match the pending spec and declared projection. [Activity core and mode extension](19-activity-core-and-mode-extension.md) is normative for this layer.

## IDs and authored data

Use small stable newtypes (`UnitId`, `EffectId`, `AbilityId`, `ActionId`) instead of strings in runtime hot paths. Keep stable human-readable keys in Excel and Sora references, then compile them into validated IDs when `combat-data` loads the exported bundle.

Ability data should describe ordered operations rather than embed scripts for every simple ability:

```rust
pub struct AbilityDef {
    pub target_rule: TargetRule,
    pub cost: Vec<ResourceOp>,
    pub phases: Vec<AbilityPhase>,
    pub energy_gain: EnergyGain,
    pub tags: TagSet,
}

pub enum AbilityOp {
    Damage(DamageSpec),
    ReduceToughness(ToughnessSpec),
    Heal(HealSpec),
    ApplyEffect(EffectApplication),
    ModifyResource(ResourceOp),
    AdvanceAction(ActionShift),
    QueueAction(QueuedActionSpec),
}
```

Native Rust handlers are appropriate for exceptional mechanics, but they should produce the same operations/events and may not mutate arbitrary state behind the resolver. Use a static registry keyed by validated stable handler IDs, not a dynamic plugin ABI or character-ID branches in shared systems. The typed program model is normative in [Rule IR and native handlers](11-rule-ir-and-native-handlers.md).

Do not expose Sora-generated row structs in `combat-core` public APIs. They are transport records at the data boundary, not domain types. This isolates combat rules from Sora schema/codegen changes.

## Stats and modifiers

Use typed stat queries and layered modifiers:

```rust
pub enum Stat {
    MaxHp,
    Attack,
    Defense,
    Speed,
    CritRate,
    CritDamage,
    BreakEffect,
    EffectHitRate,
    EffectResistance,
    EnergyRegenRate,
}

pub enum ModifierLayer {
    BaseAdd,
    PercentOfBase,
    Flat,
    FinalAdd,
    FinalMultiply,
}
```

Formula-specific collections such as DMG Boost, vulnerability, DEF ignore, and RES penetration should be evaluated through a `DamageContext`, because their applicability depends on source, target, element, ability tags, and effect state.

Avoid a universal `HashMap<Stat, Scalar>` as the only model. Even with the correct numeric representation, it loses validation, formula stage, and query context. [Modifier and snapshot pipeline](12-modifier-and-snapshot-pipeline.md) defines applicability, stacking, snapshot, and cycle rules.

## Event queue, not recursive callbacks

Triggers should convert observed events into queued work:

```rust
struct PendingReaction {
    priority: ReactionPriority,
    sequence: u64,
    source: UnitId,
    cause: EventId,
    operation: ReactionOp,
}
```

Resolve one atomic operation, append its events, collect eligible triggers in stable order, enqueue reactions, then pop the highest-priority reaction. Track `(effect_instance, trigger_id, cause_event)` to enforce once-per-event semantics.

Define hard budgets for events per command and nested reaction depth. Exceeding a budget deterministically commits a stable fault after completed atomic operations, or rolls back the uncommitted journal and then commits `Faulted` from the pre-command state. It must not return with undocumented partial mutation.

## Deterministic RNG

All randomness must pass through one injected deterministic generator. Never call thread-local randomness from a rule.

Each draw should emit or optionally trace:

- draw index;
- purpose (`Crit`, `Debuff`, `BounceTarget`, `AggroTarget`, and so on);
- candidate ordering/weights;
- raw sample and selected result.

Seed plus command stream plus data revision must reproduce the same event stream. Pin `rand = "=0.10.2"` with its `chacha` feature and identify `ChaCha8Rng` plus the project's integer mapping as `chacha8-rand-0.10.2-intmap-v1`. Disable default/thread RNG features for authoritative crates. Never use generic distribution helpers: rejection sampling, range mapping, weighted choice, no-candidate draw consumption, and stable candidate order are project-owned replay behavior. Changing any of them is a replay-format change.

## Numeric representation

The project requires cross-platform replay consistency, so authoritative simulation uses exactly pinned `fixnum` decimal fixed-point arithmetic from the first real combat implementation. The initial backing representation is signed `i64` with six decimal places. All operations are checked and every precision-losing operation names its rounding policy.

`f32` and `f64` are forbidden in authoritative configuration values, battle state, operations, events, RNG decisions, and replay hashes. A test-only `f64` reference calculator may support formula research and differential tests.

Never use a rounded display AV as state. Fixed-point `ActionGauge` and `Speed` plus stable integer sequence keys determine ordering. Represent percentage values consistently as ratios (`0.25`), not mixed whole percents (`25`). Domain newtypes hide the backing library and prevent unit mistakes.

The complete representation, rounding, RNG, collection-order, fault, and state-hash requirements are normative in [Cross-platform determinism and numeric policy](09-determinism-and-numerics.md).

## Serialization and replays

Production configuration is the Sora binary bundle. Debug JSON is for review and tests, not the shipping load path. Version all external schemas and identify the exact exported bundle:

```rust
pub struct ReplayHeader {
    pub schema_version: u32,
    pub rules_revision: String,
    pub data_revision: String,
    pub config_bundle_sha256: [u8; 32],
    pub numeric_policy_revision: String,
    pub rng_algorithm_revision: String,
    pub state_hash_revision: String,
    pub seed: u64,
}
```

A replay stores initial battle or activity identity and accepted commands. Activity replays contain nested battle identities/results under the same config/profile digest. Events and per-command hashes may be stored for audit and divergence detection but must be reproducible. Replay loading must use the same `config.sora` digest or an explicit migration/archive lookup. The canonical byte layout and planned command-line surface are specified in [Replay, CLI, and engine integration](16-replay-cli-and-engine-integration.md).

## Bevy integration later

The Bevy adapter should:

- own an `Activity` and its current `Battle`, or run them in a dedicated simulation worker;
- translate player/AI intent into offered `ActivityCommand` or battle `Command` values;
- publish `ActivityEvent` and `BattleEvent` values for flow, animation, audio, camera, and HUD systems;
- wait for presentation acknowledgments only outside the core;
- use `UnitId` mapping components to relate domain units to Bevy entities;
- render battle state from `BattleView` and flow/roster/score state from `ActivityView`; never infer authoritative state from presentation components.

Bevy time, frame rate, and entity iteration order must not affect core results.

## Validation

Validate authored data before constructing a battle or activity:

- the Sora bundle format and schema fingerprint match the generated reader;
- the config manifest's Sora CLI, data, and required rules revisions are accepted;
- all IDs resolve and references are acyclic where required;
- hit ratios and target rules are legal;
- costs and resource caps are non-negative;
- duration policies have a tick owner and phase;
- modifier stages and tags are known;
- Bounce has a positive hit count and a fallback for an empty pool;
- every triggered native handler is registered;
- Toughness values specify raw units;
- no probability or percentage uses ambiguous units;
- authoritative numeric fields parse exactly into the pinned fixed-point representation;
- every precision-losing operation has an explicit rounding policy;
- every serialized/hashed numeric value has a fixed-width canonical encoding;
- activity graphs have valid entry/terminal reachability, typed slots, bounded loops, and legal edges;
- participant pools, uniqueness, trial/borrow/draft/ban, formation, and loadout locks are coherent;
- clock, metric, objective, spawn, carry, and BattleResult projections are complete and bounded;
- mode profiles reference generic activity definitions rather than a private command processor.

Sora performs structural validation while exporting. `combat-data` remains responsible for combat/activity invariants that require understanding event order, flow graphs, projections, operations, native handlers, or cross-table behavior.

## Test strategy

Use five layers:

1. Formula unit tests: synthetic vectors for DEF, RES, CRIT, healing, shields, Break, and effect chance.
2. State-machine tests: turn start/end, death mid-hit, wave transitions, action advance, control, and queue priority.
3. Property tests: HP and resources remain in legal ranges; seeded replay is stable; invalid targets never receive effects; event IDs are monotonic.
4. Golden battle tests: a small original roster with recorded command streams and complete expected events/state hashes.
5. Activity/profile tests: scopes, participants, graph decisions, clocks/metrics, result projection, persistence, and seeded end-to-end profile hashes.

Minimum golden cases before engine integration:

- the 648/720 ordinary-damage vector from [Damage and sustain](03-damage-and-sustain.md);
- same-level DEF multiplier equals 0.5 with no modifiers;
- 100 SPD acts every 100 AV and 200 SPD every 50 AV;
- a mid-gauge Speed change preserves AG and prorates AV;
- 25% Break delay adds exactly 2,500 AG;
- multi-hit CRIT rolls are independent and ordered;
- shield overflow and background shield break;
- Effect Hit Rate calculation with specific resistance;
- each of the seven Break elements;
- follow-up action resolves before a queued normal zero-AG turn;
- replaying the same seed and commands yields an identical event hash.

## Recommended implementation order

1. Pin Sora and `fixnum`, implement domain numeric wrappers/canonical encoding, define schemas, generate Excel templates/Rust readers, load `config.sora`, validate IDs/unit stats, and add deterministic integer RNG.
2. Timeline and ordinary turns without interrupts.
3. Commands, target patterns, hits, ordinary damage, death, and waves.
4. effects, durations, DoT, healing, shields, Skill Points, and Energy.
5. Toughness and the seven Weakness Break variants.
6. interrupts, Ultimates, follow-ups, counters, and extra turns.
7. replay format and deterministic battle controller.
8. `activity-core` graph/scopes/participants/persistence/BattleSpec handoff, then the one-node Standard profile.
9. One `mode-challenge` vertical slice per classic challenge family and one `mode-universe` slice per permanent universe family.
10. Survival, boss-rush, trial/draft-roster, and continuous-spawn fixtures proving data-driven future extension.
11. CLI orchestration and coverage reporting, followed by optional engine adapters.

This order keeps every stage headless and testable.
